# 产品目录标准化迁移

本轮先不直接执行重命名，而是先输出一份精确的迁移影响清单，供你最终确认。已确认的口径如下：

- `OpenSwim Pro` 是标准产品名，`SEAL` 是代号。
- `OpenFit Pro` 是标准产品名，`NT` 是代号。
- `OpenSound` 按平台处理，标准名为 `OpenSound Platform`。
- `BOS = OpenRun Pro 2`，`LOK = OpenDots ONE`。
- `NCE` 和 `XS` 属于未上市产品，采用双名制：目录先保留代号，frontmatter 和展示层可写预定标准名。

## 本轮要做的事

### 1. 固定目录与命名映射口径

基于当前字典与已确认口径，整理出“目录名、标准产品名、代号、平台名”的唯一映射表，作为迁移影响清单的依据。

重点文件：

- [知识库及前端开发/知识库/运行域/治理文档/命名治理字典.json](知识库及前端开发/知识库/运行域/治理文档/命名治理字典.json)
- [知识库及前端开发/知识库/运行域/脚本/kb_naming_rules.py](知识库及前端开发/知识库/运行域/脚本/kb_naming_rules.py)

### 2. 生成第一批“已确认可迁移”影响清单

只列出以下对象的拟改动，不立即执行：

- `知识域/产品主张/OpenSwim-Pro/` → `知识域/产品主张/OpenSwim Pro/`
- 所有 `OpenSwim-Pro` / `OpenSwim_Pro` 文件名 → `OpenSwim Pro`
- 所有 `OpenFit-Pro` / `OpenFit_Pro` 文件名 → `OpenFit Pro`
- 所有 `OpenSound-Pro` 平台相关文件/目录 → `OpenSound Platform` 口径清单（若当前目录不存在，则只列涉及文件名与标准名）

影响清单需覆盖：

- 目录路径变化
- 文件名变化
- frontmatter 中的 `canonical_title` / `canonical_path` / `related_products`
- `source_docs` / `supersedes` 等引用链变化

重点文件：

- [知识库及前端开发/知识库/运行域/脚本/apply_kb_naming_migration.py](知识库及前端开发/知识库/运行域/脚本/apply_kb_naming_migration.py)
- [知识库及前端开发/知识库/运行域/治理文档/审计记录/kb_naming_migration_preview.md](知识库及前端开发/知识库/运行域/治理文档/审计记录/kb_naming_migration_preview.md)

### 3. 为代号目录生成“待确认但不迁移”的影响清单

对以下对象采用“只出建议，不进入第一批自动迁移”的策略：

- `NCE`：目录保留代号，但 frontmatter / 展示层使用预定标准名
- `XS`：目录保留代号，但 frontmatter / 展示层使用预定标准名
- `BOS`：列出其迁移到 `OpenRun Pro 2` 的建议清单，但不自动执行，等你看完再决定
- `LOK`：列出其迁移到 `OpenDots ONE` 的建议清单，但不自动执行，等你看完再决定

这里会把“目录策略”和“展示策略”分开，避免未上市产品过早把代号目录洗掉。

### 4. 输出一份最终确认表

最终交付一份你可以逐项勾选的确认表，分成三栏：

- 立即可执行
- 待确认
- 仅改展示不改目录

## 执行边界

本轮不做实际文件迁移，只做影响清单与确认表。

## 风险控制

- 不直接动 `NCE / XS` 目录，避免未上市产品目录提前公开化。
- `OpenSound` 不按产品归类，避免平台名被错误洗成单一 SKU。
- 所有实际迁移前，必须先看到 rename map 和引用链影响。
