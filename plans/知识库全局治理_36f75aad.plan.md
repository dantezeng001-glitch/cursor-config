# 知识库全局治理方案

## 目标

把 `知识库/` 从“已有规则但落地不彻底”的状态，升级为一套真正闭环的系统：

- 存量文档能批量重命名、重分类、修复引用。
- 新导入文档一入库就按标准命名，不再先脏后补。
- 前端不再把路径变更当成数据丢失，旧链接、评论、来源追溯都能连续。
- 用户在 `知识域/` 中看到的目录与文件名，能直接体现知识角色、产品归属和内容状态。

## 现状判断

当前基础设施已经具备，但“规则宽松 + 迁移未闭环”导致治理看板比真实体验更乐观：

- `[C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\脚本\kb_naming_rules.py](C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\脚本\kb_naming_rules.py)` 已能推断 `canonical_path`、`recommended_action`、`auto_apply_eligible`，但对很多用户视角下“仍不统一”的文件仍给出高分或仅建议补元数据。
- `[C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\前端开发\kb_backend.py](C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\前端开发\kb_backend.py)` 目前直接用相对路径作为文档主键，评论也按 `pagePath` 存储；只要真实改名，就会牵动详情页、书签、评论和 `source_docs` 解析。
- `[C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\同步文档\index.md](C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\同步文档\index.md)` 仍暴露旧来源标题、过程态 sheet 名和历史命名痕迹，说明“新文档入库即规范化”还没真正执行。

## 治理原则

1. `知识域/` 的真实目录和文件名作为标准真相，前端只做兼容和可视化，不再反向替代治理。
2. 产品类目录统一使用正式产品名，代号如 `NCE/XS/LOK` 退到别名、元数据和搜索召回层。
3. `source_summary` 与“正式知识页”分离治理：来源入口页可以保留 `_同步版`，但知识正文、卖点卡、原理页、判断页全部转为自然语言标准名。
4. 所有批量迁移必须同时产出 rename map，并同步修复 `source_docs`、`supersedes`、前端旧路径访问和评论键。
5. 新导入文档默认走同一条规范化流水线，禁止再把 `待锁定`、`(2)`、`持续更新`、代号目录等历史噪音直接带入主库。

## 实施方案

### 第一阶段：收紧“标准命名与归类规则”

更新治理规则与字典，先把“什么算统一”定义得足够清楚。

重点调整：

- 在 `[C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\脚本\kb_naming_rules.py](C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\脚本\kb_naming_rules.py)` 中重写产品目录推断、命名检查和建议路径生成逻辑。
- 在 `[C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\治理文档\命名治理字典.json](C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\治理文档\命名治理字典.json)` 中把正式产品名定义为主目录名，并补一套 codename -> official name 映射。
- 把以下命名差异从“可接受”升级为“需迁移”：
  - 产品目录代号 `NCE/XS/LOK`
  - 文件名中的过程态词 `待锁定/加工中/持续更新`
  - 空格版  `同步版`、括号副本、长噪音标题
  - 同类文档中英文/下划线/空格混用
- 明确两类命名模板：
  - 正式知识页：`知识域/<知识区>/<子类>/<自然语言标准名>.md`
  - 来源入口页：`知识域/<知识区>/<子类>/<自然语言标准名>_同步版.md`

### 第二阶段：建立“安全迁移层”而不是直接硬改

先做迁移资产，再执行重命名，避免一改路径就把前端和评论打断。

需要新增或扩展：

- rename map / alias map：记录 `old_path -> new_path`、`doc_id`、历史别名。
- 评论迁移策略：把 `[C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\前端开发\data\comments.json](C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\前端开发\data\comments.json)` 从“只认 pagePath”升级为可按新路径回写或批量迁移。
- 引用修复器：批量更新全库 Markdown frontmatter 中的 `source_docs`、`supersedes`、`canonical_path`、`original_path` 等字段。
- 迁移预览与回滚资产：继续输出预览清单，但这次必须真正包含“路径会改成什么、哪些引用会被改、哪些评论会被迁”。

### 第三阶段：把“新文档入库”改成强制规范化流程

把规范前移到入口，避免未来继续积累命名债务。

需要梳理和收口的入口：

- `收件箱 -> 归档`
- `同步二进制文档 -> 运行域/同步文档`
- `同步产物 -> 知识域来源入口页`

建议做法：

- 让归档/同步脚本在落盘前就生成正式产品名、语义化文件名和标准 frontmatter。
- 对 Excel/PPT/PDF 抽取结果统一执行“去过程态名、去副本痕迹、保留 fileID 目录、语义化正文文件名”。
- 对新建来源入口页自动写入 `canonical_title`、`canonical_path`、`original_*`、`source_docs`。
- 对无法高置信判断归类的文件，进入明确的待确认队列，而不是直接以旧名入主库。

这里需要联动的脚本通常会落在：

- `[C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\脚本](C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\知识库\运行域\脚本)`
- 尤其是架构说明中已列出的归档、同步、刷新索引相关脚本。

前端需要从“路径驱动页面”升级为“知识对象驱动页面”。

重点改造：

- 在 `[C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\前端开发\server.py](C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\前端开发\server.py)` 的详情接口上支持“旧路径访问新文档”。
- 在 `[C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\前端开发\static\app.js](C:\Users\016551\OneDrive\Desktop\科技树\知识库及前端开发\前端开发\static\app.js)` 中把页面展示重点从“当前路径”转到“标准标题 + 所属知识区 + 归类说明 + 历史别名”；并在打开旧路径时自动跳转到新路径。
- 让评论、来源追溯、相似主题跳转都基于稳定对象而不是单纯路径字符串。
- 视情况优化 `markdown.js` 的内部链接处理，让库内相对链接在重命名后仍能落到正确页面。

## 预期交付

1. 一套更严格的命名与归类规则，正式产品名成为主目录标准。
2. 一套可执行的迁移脚本与迁移映射文件，能批量处理存量文档并修复引用。
3. 一套新文档导入规范化流程，保证增量不再返工。
4. 一套前端兼容机制，保证旧链接、评论、治理看板、来源追溯在重命名后继续可用。
5. 一批更新后的治理文档，使后续维护者能看懂规则、流程和边界。

## 实施顺序建议

1. 先改规则和字典，不急着立刻改文件。
2. 再产出真实迁移预览和 alias/rename map。
3. 小批量试迁移 `产品主张` 与 `术语卡`，验证评论、来源链和前端跳转。
4. 验证通过后再批量迁移 `知识地图`、`同步文档`、`策略判断`。
5. 最后把新导入流程切到强制规范化模式，防止回潮。
