## 一、整体判断（先认这 6 行骨架）

1. 大方向是过关的：[static/index.html](前端开发/static/index.html) 已经做了 `skip-link / sr-only / aria-live / focus-visible / nav main`，不是粗糙稿。
2. 5 类高频体验问题：长 Markdown 表格在窄屏溢出、详情页元数据 7 项偏多遮蔽正文、搜索/筛选状态不入 URL（不能分享/收藏/后退）、文档列表无虚拟化（接近 100 项时滑动卡）、按钮文案过虚（"应用筛选"/"搜索"/"返回浏览"）。
3. 3 类小但显眼的 polish：`...` → `…`、直引号 → 弯引号、按钮文案 → Title Case + 具体动作。
4. 1 个潜在重构：手写的 [static/markdown.js](前端开发/static/markdown.js) 表格识别脆弱，长期建议换 `marked + DOMPurify`（独立任务，不放进本轮）。
5. 不要做的事：不要重写整套 CSS。当前米色 + Fraunces 衬线"档案风"是差异化，丢了等于变成普通 dashboard。
6. 范围说明：本轮按你选的 D（带产品体验维度）。审计严格按 Vercel guidelines；体验那部分是我的判断，会标"我的判断"。

## 三、产品体验问题（我的判断，按"团队跨部门低门槛"评估）

### 必修（影响"第一次能不能用起来"）

E1 详情页元数据 7 项过载：[前端开发/static/app.js:587-616](前端开发/static/app.js) 一上来就把"稳定文档 ID / 当前归档路径 / 负责人 / 更新时间 / 相关产品 / 相关概念 / 替代链"全摊开，非治理人员看不懂"docId / canonicalPath / supersedes"。我的建议：默认只显示"负责人 / 更新时间 / 相关产品"3 项，其余收进 `<details>` 标签"治理元数据"。

E2 顶部全局搜索没用上模糊匹配体验：[前端开发/static/app.js:1087-1091](前端开发/static/app.js) 提交后跳转到 `/search?q=...`，没有"输入即时显示候选"的 typeahead。我的判断：跨部门用户来了不会先去"浏览"，会直接在搜索框打 `OpenRun Pro 2`，希望 200ms 内看到下拉。本轮不做完整组件，但可以在搜索框失焦/Enter 才触发不变，在输入框下加 1 行轻提示"按 Enter 搜索"。

E3 浏览页空状态太弱：[前端开发/static/app.js:411](前端开发/static/app.js) 命中 0 文档时只显示 `<empty-state-template>`，没说"是哪个筛选导致 0 命中、点哪能放宽"。建议：列出当前已选条件（zone / kbType / status / owner / query），每条带"移除"按钮。

### 建议（明显提升）

E4 详情页"原始 Markdown"折叠区视觉太抢戏：[前端开发/static/app.css:776-785](前端开发/static/app.css) `.raw-markdown` 用深色背景，整段折叠展开后比正文还长，喧宾夺主。建议默认折叠收紧，添加"复制原始 Markdown"按钮替代直接展示一大段黑底文字。

E5 首页"按知识区进入" + "按知识类型进入" + "最近更新" 三块卡组拥挤：[前端开发/static/app.js:331-349](前端开发/static/app.js) 都是浏览入口，新人要做选择题。建议合并成一块"按维度进入"+ Tab 切换（知识区 / 知识类型）。但这是设计决定，不强求。

E6 hover/focus 视觉对比度不够明显：[前端开发/static/app.css:457-463](前端开发/static/app.css) hover 仅改 border-color + background，可读性提示弱。规则要求"hover/active/focus 比 rest 更突出"。我的建议：hover 时给卡片加一条左侧 3px 品牌色 ribbon 或 `transform: translateY(-1px) + box-shadow`（同时尊重 `prefers-reduced-motion`）。

### 可选（锦上添花）

E7 缺少键盘快捷键：跨部门用户不太用快捷键，所以这一项是"可选"。但 `/` 聚焦搜索是行业惯例（GitHub / Vercel / 文档站都用），ROI 高、成本低。

E8 顶部 nav 缺面包屑回溯：详情页内已有面包屑 `首页 / 知识区`，但浏览页和搜索页都没。可以在 nav 下加一条 breadcrumb 区。

E9 详情页缺"上一篇 / 下一篇"：长尾改进。

## 四、ROI 最高的 5 项（如果只做 5 件事，做这些）

1. **必修 E1（元数据简化 + details 收纳）**：本轮收益最大，跨部门人才看得懂详情页。涉及 [前端开发/static/app.js:587-616](前端开发/static/app.js)。
2. **必修 E3（浏览页空状态展示已选条件 + 移除按钮）**：避免"我点了一堆筛选然后看到 0 篇，怎么退出来"的死循环。涉及 [前端开发/static/app.js:411](前端开发/static/app.js)。
3. **合规 + Polish 集合修**：`...` → `…`、按钮文案具体化、`Intl.DateTimeFormat`、`#app-status` 真正用起来、图片 `loading="lazy" + aspect-ratio`、`translate="no"`。这一组耗时少、合规率提升大。涉及 [前端开发/static/app.js](前端开发/static/app.js)、[前端开发/static/markdown.js](前端开发/static/markdown.js)、[前端开发/static/app.css](前端开发/static/app.css)。
4. **建议 E4（"原始 Markdown" 改成"复制按钮"）**：减少视觉噪音。涉及 [前端开发/static/app.js:627-630](前端开发/static/app.js)、[前端开发/static/app.css:776-785](前端开发/static/app.css)。
5. **建议 E6（hover/focus 对比度提升 + reduced-motion 兜底）**：最低成本拿到"专业感"。涉及 [前端开发/static/app.css:457-463](前端开发/static/app.css) + 顶部新增一段 `@media (prefers-reduced-motion: reduce)`。

剩下的（虚拟化、typeahead、首页布局合并、快捷键、面包屑、上下篇）放后续。
