# Content Integrity Guard — Reference

补充 `SKILL.md` 的详细素材：关键词扫描包、修订措辞模板、膨胀检测、审计日志模板。需要时按需 Read，不需要每次都读。

## Scan keyword pack

分三组。需要用 `Grep` 工具扫描目标目录（`-n` 显示行号，`output_mode: "content"`）。

### Group A — Rule-masquerading words（措辞升格嫌疑）

```
按.*规范|按.*纪律|按.*节奏|符合.*规范|符合.*节奏|符合.*规定|遵守.*纪律|系列规范|硬性规定|硬性约束|一律|不得|必须|凡.*必须|任何.*必须|行业通行|标准做法|最佳实践|惯例|约定俗成|隐蔽技巧|天然抗性|免疫度|免疫度极高
```

### Group B — Audience / algorithm claims（无出处的受众行为断言）

```
LinkedIn.*受众会|观众.*会|读者.*会|用户.*会注意|算法.*会|算法.*压制|算法.*推|会让.*更愿意|更倾向于|自报家门|会让.*评论|会触发
```

### Group C — Self-referential authority（把自己伪装成共识）

```
我们.*坚持|我们.*一直|我们.*规定|我们.*要求|这是.*做法|我们的.*原则|这是为了.*设计|这是标准|Shokz.*一向|品牌一贯
```

### Group D — Bloat detectors (信息凝练 第三维度专用)

```
非常|极其|实际上|事实上|其实|说白了|总的来说|一般来说|某种程度上|大体上|基本上|某种意义上|在.*这一节里|正如上文|如前所述|请注意|值得强调|重要的是|需要指出|需要提醒|值得一提
```

### Group E — Quantitative drift candidates（数字漂移）

不用预设关键词——要求对每份已 touched 的文档运行一次"抽取所有数字 + 交叉对照"。具体做法：

1. 用 `Grep` 配 `\d+` 找出所有数字，带上 -B 1 -A 1 获取上下文
2. 按上下文分组：年数 / 国家数 / 样本数 / 百分比 / 金额 / 时长
3. 同组内比对：同一事实的数字在不同文件里是否一致

## Remediation phrasing pack

每条 🔴 红灯 / 🟡 黄灯的修订都遵循"**保留洞察，重定位认知地位**"的原则。不要删除，要降格。

### 红灯修订模板

**Before（危险）**：

> LinkedIn 受众对"自报家门式"的品牌露出有天然抗性。

**After（安全）**：

> **这是我的写作判断，不是规划硬性规定**——依据来自 [具体文档 + 章节]，该文档结论是 [具体结论]。但 **"…有天然抗性"这个具体断言我并没有量化数据支持**，它是从 [文档] 推导出来的写作直觉。

---

**Before**：

> 这是 LinkedIn 上 thought leadership 写作的一个隐蔽技巧。

**After**：

> **这是我个人的写作习惯，不代表行业通行技巧**。

---

**Before**：

> LinkedIn 算法对"看起来像广告"的文字帖有压制。

**After**：

> **我的判断（无 LinkedIn 官方公开算法数据直接支持）**：……这是 LinkedIn 营销社区的普遍说法，[具体文献参考编号] 有间接佐证，但我没有 [本账号] 的实测数据。

---

**Before**：

> 科技媒体和测评人会注意到这句话。

**After**：

> **"…会注意到这句话"是我的期望，不是事实**——真实引用率要靠发布后数据回证。

### 黄灯修订模板

**Before**：

> 任何素材在对外使用之前，都必须通过以下三层核查。

**After**：

> 在对外使用素材之前，**我建议**（非公司级强制流程、非已立项的合规规范）做以下三层核查。

---

**Before**：

> 对每一项素材，**我们的用法原则**是：

**After**：

> 对每一项素材，**我（[明确的主体，如"营销侧"/"产品侧"]）建议遵循的用法原则**是（可被 [明确的有权推翻方] 推翻）：

---

**Before**：

> 这会直接损伤这波传播的可信度基础。

**After**：

> **我的风险判断是**：**可能**损伤……的可信度基础。（损伤的幅度我无法实证预测，但 [列出一般性行业逻辑]。）

### 切勿这样修订

- ❌ 不要只加一个"可能"就当改好了——"可能损伤可信度"仍然是同样的断言，只是加了安全词
- ❌ 不要删除原句再重写——会丢失洞察
- ❌ 不要把 🔴 硬改成 🟢 通过捏造一个"据研究表明"的出处

## Cross-document fidelity workflow

针对维度 2（信息不失真）的专用扫描流程：

1. **列出已 touched 的所有文件**（本次 session 内创建 / 修改过的所有文档）
2. **为每个重要实体建立对照表**：
   - 产品名：`OpenFit Pro` vs `Open Fit Pro` vs `开适 Pro`
   - 商标：`DirectPitch™` vs `DirectPitch` vs `Direct Pitch`
   - 数字：`5 大洲` vs `five continents` vs `全球` —— 是否同源、单位一致
   - 时间跨度：`三年` vs `2022-2025` vs `约 3 年`
   - 金句：原句必须逐字一致，不能改写
3. **逐行比对**，任何不一致都列入审计日志
4. **不要自行判断"哪个版本是对的"**——交给用户或 authoritative source 决定
5. **修正时全量替换**，避免只改一处

## Bloat detectors (维度 3 详细清单)

### 常见可删除句式

| 类型 | 示例（粗体部分可删） | 删后效果 |
|---|---|---|
| 元叙述 | **在下面这一节里我们将讨论** 拆条 1 的配文 | 拆条 1 的配文 |
| 自述性限定 | **应该说** 这个方案是有风险的 | 这个方案是有风险的 |
| 情绪化强调 | **非常** 重要的 **极其** 关键的一点 | 重要且关键的一点 |
| 过度 hedging | 这 **可能在某种程度上 或许** 会影响 | 这可能会影响 |
| 重复 | 这是一条很 **好** 的建议，**不错**，**值得参考** | 这是一条值得参考的建议 |
| 冗余列举 | A、B、C **等等** | A、B、C |
| 套话 | **众所周知** / **毋庸置疑** / **说白了** | 直接删 |

### 段落级膨胀测试

对每段问这三个问题：

1. 删掉这一段，上下文是否还成立？→ 成立就删
2. 这一段是否只是把上一段换个说法？→ 是就删
3. 这一段添加的"事实 / 论证 / 证据"能用一句话概括吗？→ 能就压到一句话

### 何时 **不应该** 压缩

- 法务条款、合规免责声明：精确性 > 凝练
- 面向不熟悉领域的读者的首段解释：可读性 > 凝练
- 金句 / 口号 / 记忆点：节奏和韵律 > 字数
- 引用原文：不得改动

## 对应 AI 助手的执行节奏建议

- **小任务（单份文档）**：不强制触发本 skill，但写完后快速过一次 self-check 7 条
- **中任务（2-3 份文档）**：完稿前必须扫一次 Group A + B
- **大任务（≥ 4 份文档 / 有金句 / 有跨文档引用）**：完稿前必扫所有 5 组 + 建立 cross-doc fidelity 对照表 + 沉淀审计日志

**什么时候值得主动提醒用户做 v{N+1}**：

- 新增 ≥ 2 份文档到已审计的项目目录
- 更新了金句 / 产品名 / 核心数字
- 用户再次质疑某处措辞的出处（本 skill 就是这么诞生的）
