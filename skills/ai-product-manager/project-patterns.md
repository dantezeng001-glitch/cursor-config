# 项目设计参考模式

以下 7 个模式从实际项目中提炼。**设计新项目时逐条过一遍，适用的直接套用；评审现有项目时当作 checklist 检查缺了哪个。**

## 模式 1：数据源与交付物分离

| 要素 | 说明 |
|------|------|
| 核心思想 | 数据存机器可读格式（JSON/YAML），交付物（Excel/PDF/网页）由脚本生成 |
| 好处 | 防止"反向编辑交付物导致数据丢失"；数据可被多个脚本复用；版本变更可 diff |
| 反模式 | 直接在 Excel 里维护数据，脚本从 Excel 读 — 格式与数据耦合 |
| 实例 | 统一表述知识库：`knowledge_base.json`（数据源）→ `generate_excel.py`（生成）→ `Shokz统一表述知识库.xlsx`（交付物） |

## 模式 2：收件箱模式（低门槛输入）

| 要素 | 说明 |
|------|------|
| 核心思想 | 提供一个"扔进去就行"的入口，后续解析、分类、归档由 AI/脚本完成 |
| 好处 | 非技术用户零学习成本；输入与处理解耦；可批量处理 |
| 适用场景 | 任何需要收集非结构化输入再转化为结构化数据的流程 |
| 关键设计 | 收件箱里放一个说明文件；处理完后源文件归档到正式目录，收件箱保持清空 |

## 模式 3：人机断点协作

| 要素 | 说明 |
|------|------|
| 核心思想 | AI 做体力活（提取/比对/生成），人只在关键决策点介入（确认/修改/拒绝） |
| 好处 | 效率最大化的同时保留人的判断权；出错可追溯到具体断点 |
| 设计要点 | 明确哪些步骤是 AI 自动、哪些步骤需要人确认；断点处必须输出清晰的变更清单供人决策 |
| 反模式 | 全自动无确认（不可控）；每步都要人确认（太累） |

## 模式 4：Cursor Rules 作为运行时记忆

| 要素 | 说明 |
|------|------|
| 核心思想 | 把项目的工作流规则、文件导航策略写进 `.cursor/rules/`，AI 每次对话自动加载 |
| 好处 | AI 不需要重新学习流程；新对话也能无缝衔接；规则随项目版本演进 |
| 设计要点 | 规则文件聚焦"怎么做"而非"是什么"；路径引用必须与实际目录同步更新 |

## 模式 5：生成前自动备份

| 要素 | 说明 |
|------|------|
| 核心思想 | 每次覆盖写入前，脚本自动将旧版移至 `历史版本/` 并加时间戳 |
| 好处 | 无需手动备份；可随时回溯；不依赖 Git |
| 设计要点 | 备份目录与输出目录同级；文件名带版本号或日期；定期清理过旧备份 |

## 模式 6：来源可追溯

| 要素 | 说明 |
|------|------|
| 核心思想 | 每条数据标注出处（哪个文件、哪个环节），不猜测、不编造 |
| 好处 | 有争议时可回源核对；建立数据信任；新人可自行验证 |
| 反模式 | "大概是从 XX 来的" — 不确定的来源等于没有来源 |

## 模式 7：命名规范先行

| 要素 | 说明 |
|------|------|
| 核心思想 | 项目启动时即定义文件/文件夹命名规则，写入 README |
| 好处 | 文件量增长后仍可保持视觉清晰；新人无需问"这个文件叫什么" |
| 设计要点 | 新文件强制遵守、历史文件宽容；规则简单到能一句话说清 |
